# Hy3 Research Studio 技术报告

## 一、项目概述

**项目名称**：Hy3 Research Studio — AI 原生科研与创作工作台

**项目定位**：基于腾讯 Hy3 大语言模型的端到端科研与创作平台，深度融合文献检索、智能分析、写作辅助、AI 微应用创建等能力，覆盖从想法提炼、文献综述、论文研讨到自定义 AI 功能构建的完整科研工作流。

**技术栈**：
- 后端：Python 3 + FastAPI + SSE 流式响应 + JSON 文件持久化
- 前端：原生 HTML/CSS/JavaScript SPA（无框架依赖，轻量可部署）
- AI 层：Hy3 API（OpenAI 兼容接口），覆盖规划、检索增强、内容生成、JSON 结构化输出、工具调用（Function Calling）
- 检索层：OpenAlex / Crossref / arXiv（免 Key 开箱即用）+ Semantic Scholar（可选增强）
- 文档解析：pypdf 解析 PDF 文献

## 三、核心技术方案

### 3.1 深度研究流水线（Research Pipeline）

这是系统最核心的端到端流程，采用 8 阶段管线设计：

**Stage A — 研究规划（Planning）**
- 调用 Hy3，将用户研究主题拆解为 2-5 个子问题（SubQuestion），每个子问题附带英文检索式
- 生成报告大纲（report_outline）和关键分析维度
- 产出：`ResearchPlan` 对象

**Stage B — 多源检索（Evidence Search）**
- 基于规划结果，为每个子问题并行检索 OpenAlex/Crossref/arXiv
- 支持用户上传文档（PDF/TXT/MD）和文库选中资料作为额外证据
- 跨源去重，统一编号 s1, s2, s3...
- 快速/标准/深度三档控制每子问题检索量（4/6/8 篇），指定引用数时自动放大
- 产出：`SourceDocument[]` 列表

**Stage C — 证据压缩（Evidence Compression）**
- **核心差异化设计**：将每子问题的原始文献片段（摘要+Snippet）交给 Hy3 压缩为结构化证据包（EvidencePacket）
- 每个 Packet 包含：主题、核心结论、方法、实验设置、局限性，以及支持文献引用
- 压缩比通常在 5:1~10:1，大幅降低后续推理上下文量
- 产出：`EvidencePacket[]` 列表 + 压缩比指标

**Stage C2 — 假设生成（Hypotheses）**
- 基于压缩后的证据包，由 Hy3 生成 3-5 条可证伪研究假设
- 每条假设附带：证据支撑、置信度（0-1）、可验证性说明

**Stage C3 — 证据图谱（Evidence Graph）**
- 识别证据包间的逻辑关系：支撑（support）、矛盾（contradict）、延伸（extend）、特化（specialize）
- 产出：`EvidenceEdge[]` 证据关系图

**Stage C4 — 研究空白（Research Gaps）**
- 基于证据包、假设、证据图谱，挖掘现有研究的不足与空白
- 每条空白附带：原因分析、建议方向、相关假设

**Stage C5 — 实验设计（Experiments）**
- 针对研究假设和空白，生成验证实验方案
- 包含：方法、数据集、评测指标、基线、预期结果

**Stage D+E — 报告生成与引用锚定（Report Writing + Citation Grounding）**
- 按规划大纲逐章流式生成，每章独立调用 Hy3 chat_stream
- 每章生成后自动正则提取 `[sN]` 引用标记，回填到章节元数据
- 支持章节级润色：扩写、缩写、改风格、加引用、驳论式写作

**关键时序指标**：
- TTFP（Time To First Plan）：规划完成时间
- TTFE（Time To First Evidence）：首批检索结果时间
- TTFR（Time To First Report）：首章报告时间
- TTR（Time To Result）：全流程完成时间

### 3.2 Agent 式检索增强（Advisor/Idea Refiner）

思路提炼模块采用 Agent + Tool Calling 架构，核心设计原则是「**先检索、再回答，绝不凭记忆编造文献**」：

1. **强制工具调用**：通过 `tool_choice={"type":"function","function":{"name":"retrieve_papers"}}` 强制 Hy3 在回答前必须先调用检索工具
2. **智能判断是否需要检索**：简单寒暄/致谢/极短消息跳过检索，避免无意义 API 开销
3. **中文检索式自动改写**：检测到中文查询时，先由 Hy3 将其重构为 2-5 个英文核心关键词（学术库以英文索引为主），首次未命中自动用英文重构式重试
4. **两层相关性过滤**：
   - 第一层：关键词规则过滤，丢弃标题/摘要不含核心词的错语料
   - 第二层：LLM 语义过滤，将用户问题+候选文献传给 Hy3，让模型逐篇判断相关性，仅保留真正相关文献并重新编号 r1, r2...
5. **收敛机制**：3 轮以上对话后注入收敛提示，引导对话从开放式讨论收口为可执行研究方案
6. **后续选择题提取**：回答生成后，由 Hy3 分析对话状态，提取 2-4 个引导下一步的选择题

### 3.3 智能检索（Smart Search）

- 用户输入主题后，Hy3 先规划出多条精准检索式（steps 1-6 条）
- 并行检索后由 Hy3 生成带 [rN] 引用的结构化检索报告
- 全流程 SSE 流式推送：规划→检索进度→来源列表→报告逐字输出

### 3.4 创造工坊（AI Feature Workshop）

**核心概念：AI Native Feature**。用户创建的不是 Prompt，而是一个拥有独立 UI、AI 能力、数据状态的微应用。

**两种创建模式**：
- **AI Quick Build（零门槛）**：一句话描述需求，Hy3 30 秒内自动生成完整功能（名称、emoji、描述、分类、系统 Prompt、布局类型、引导语、UI 配置）
- **Visual Builder（可视化设计）**：多轮对话明确需求→Hy3 生成可拖拽组件布局→预览→保存

**独立工作空间架构**：
- 每个功能通过 hash 路由 `#/feature/{id}` 进入专属工作区
- 根据 `layout_type` 加载不同布局模板（research/review/meeting/medical/chat 等）
- 退出时正确清除状态，支持反复进入

**精选模板（4 个官方模板，少而精）**：
1. **Research Workspace** 📚：三栏科研工作区（文献库+阅读区+笔记引用）
2. **Review Workspace** 🔍：审稿工作区（评分维度+审稿意见输出）
3. **Meeting Assistant** 🎙️：会议助手（转写记录+纪要行动项双栏）
4. **Medical Report** 🏥：医学报告助手（患者信息录入+病历报告双栏）

**社区生态能力**：收藏、评分（1-5 星加权平均）、Fork/Remix、使用统计、分类浏览（10 个类别）

### 3.5 Hy3 Client 封装

对 OpenAI 兼容接口的精简封装，提供四种调用模式：

| 方法 | 用途 | 特点 |
|------|------|------|
| `chat()` | 同步非流式 | 返回完整文本，自动累计 token |
| `chat_stream()` | 流式输出 | SSE 逐字推送，启用 `stream_options.include_usage` |
| `chat_json()` | 结构化输出 | 调用后自动解析 JSON，支持 ```json fenced code 块和大括号容错提取 |
| `chat_message()` | 工具调用 | 返回原始 Message 对象，支持 force_tool_name 强制工具调用 |

**JSON 解析鲁棒性**：模型返回 JSON 时可能用 markdown 代码块包裹、前后有散文说明，`_extract_json()` 实现三级解析策略（直接 parse → fenced 提取 → 平衡括号块匹配）。

### 3.6 流式响应（SSE）

所有耗时操作（研究流水线、思路提炼对话、论文对话、智能检索、功能聊天、工具调用）均采用 Server-Sent Events 推送：

- 统一事件格式：`event: <name>\ndata: <json>\n\n`
- 事件类型：`status/plan/evidence_sources/evidence_packets/hypotheses/evidence_graph/gaps/experiments/section_start/section_delta/section_done/done/error` 等
- 前端通过 `EventSource` 或 `fetch` + `ReadableStream` 消费
- 进度条阶段：planning→searching→compressing→hypothesizing→graphing→gapping→experimenting→writing→done

### 3.7 两层用户个性化卡片

```
Level 1（即时信号）：最近检索词、最近研究主题 → 捕捉当下关注点
Level 2（稳定画像）：用户在设置中声明的身份/机构/研究兴趣 + 从行为沉淀的派生兴趣 → 长期基线
```

推荐和"猜你想搜"结合两层信号，使推荐既贴合当下、又不脱离长期方向。推荐结果缓存 5 分钟，多主题并行检索时显著降低延迟。

### 3.8 数据持久化

采用 JSON 文件持久化（无外部数据库依赖，开箱即用）：

| 文件 | 存储内容 |
|------|----------|
| `data/settings.json` | 用户设置、个人画像、默认参数 |
| `data/library.json` | 文献库条目、文件夹结构 |
| `data/folders.json` | 文件夹层级 |
| `data/features.json` | 创造工坊功能列表、收藏夹 |
| `data/feedback.json` | 用户反馈条目、投票数据 |
| `data/studio_history.json` | 写作工坊历史 |
| `data/paper_history.json` | 论文研讨历史 |
| `data/advisor_history.json` | 思路提炼历史 |

所有写操作通过线程锁（`threading.Lock`）保证并发安全。

### 3.9 离线降级策略

当 `HY3_API_KEY` 未配置时，系统不会崩溃，而是进入离线模式：
- 创造工坊：启发式布局推断（根据描述关键词匹配 layout_type），返回可用功能
- 思路提炼：固定引导语提示用户描述需求
- 各模块仍可浏览界面，有清晰的"未配置"提示

## 五、前端设计

### 5.1 技术选型

- **零框架**：原生 Vanilla JS + HTML + CSS，无 React/Vue 依赖，单文件部署
- **Hash 路由**：通过 `location.hash` 和 `hashchange` 事件实现 SPA 导航，功能工作区使用 `#/feature/{id}` 独立路由
- **SSE 消费**：使用原生 `EventSource` 和 `fetch + ReadableStream` 两种模式处理流式响应
- **CSS 变量体系**：定义 `--bg-primary/--accent/--border/--text-primary` 等设计 token，支持主题切换

### 5.2 视图切换机制

统一的 `switchView(name)` 函数管理视图切换：隐藏所有 `.view`，显示目标 `#view-{name}`，并触发该视图的渲染逻辑。侧边栏导航通过 `data-view` 属性绑定，active 状态自动同步。

### 5.3 功能工作区（Feature Workspace）

每个功能拥有专属布局容器（`.fw-layout`），进入时：
1. 给 `body` 添加 `fw-mode` 类，隐藏主导航
2. 调用后端 `/api/features/{fid}/use` 记录使用
3. 根据 `layout_type` 激活对应布局面板
4. 渲染引导语（starters）
5. 绑定独立对话输入

退出时：清除 fw-mode、重置 fwState、清除 hash（使用 history.replaceState 避免触发 hashchange 冲突）。

## 六、Hy3 在系统中的角色

Hy3 贯穿系统所有 AI 能力，承担以下角色：

| 角色 | 模块 | 具体任务 |
|------|------|----------|
| **Planner（规划师）** | 深度研究、智能检索 | 拆解研究问题为子问题、生成检索式、规划报告大纲 |
| **Search Query Optimizer（检索优化器）** | 思路提炼、Advisor | 中文→英文检索式改写、关键词提取 |
| **Evidence Compressor（证据压缩器）** | 深度研究 | 将原始文献摘要压缩为结构化证据包 |
| **Reasoner（推理器）** | 深度研究、思路提炼 | 生成研究假设、识别证据关系、挖掘研究空白 |
| **Experiment Designer（实验设计师）** | 深度研究 | 生成验证实验方案 |
| **Report Writer（报告撰写）** | 深度研究、智能检索 | 逐章流式生成带引用的学术报告 |
| **Refiner（润色器）** | 深度研究 | 章节扩写/缩写/改风格/加引用 |
| **Paper Analyst（论文分析师）** | 论文研讨 | 回答关于论文内容的问题、提取大纲和创新点 |
| **Writing Assistant（写作助手）** | 写作工坊 | 生成摘要、大纲、段落 |
| **Research Coach（研究教练）** | 思路提炼 | 多轮对话引导用户想清方向，强制检索证据后回答 |
| **Feature Generator（功能生成器）** | 创造工坊 | 一句话需求→完整 AI 微应用（名称/图标/Prompt/布局/UI） |
| **Feature Designer（功能设计师）** | 创造工坊 | 多轮对话理解需求、生成可拖拽组件布局 |
| **Relevance Judge（相关性裁判）** | 检索模块 | 语义过滤检索结果，判断文献是否真正相关 |
| **Suggestion Engine（推荐引擎）** | 首页、猜你想搜 | 基于用户画像生成检索建议、推荐文献 |
| **Meta-Analyzer（元分析器）** | 思路提炼 | 对话后提取引导下一步的选择题 |

## 七、部署方案

### 7.1 环境要求

- Python 3.10+
- 依赖：`fastapi`, `uvicorn`, `openai`, `httpx`, `pydantic`, `python-multipart`, `pypdf`
- Hy3 API Key（通过环境变量 `HY3_API_KEY` 或 `.env` 文件配置）

### 7.2 启动方式

```bash
# 方式一：便捷启动
python run.py
# 访问 http://localhost:8731

# 方式二：直接 uvicorn
python -m uvicorn backend.main:app --host 0.0.0.0 --port 8731
```

### 7.3 配置项

所有配置通过 `.env` 文件管理：

- `HY3_API_KEY`（必填）：Hy3 API 密钥
- `HY3_BASE_URL`：API 端点，默认腾讯 TokenHub
- `HY3_MODEL`：模型名称，默认 `hy3`
- `DEFAULT_SOURCES`：检索源，默认 `openalex,crossref,arxiv`
- `S2_API_KEY`（可选）：Semantic Scholar API Key，增强召回

## 九、端到端 Demo 流程

### Demo 1：深度研究 — RAG 上下文压缩综述

1. 在深度研究页面输入主题："面向 RAG 的长上下文压缩方法综述"
2. 选择：文献综述 / 学术风格 / 标准深度 / 中文 / 期望引用 10 篇
3. 系统 8 阶段自动执行：规划子问题→多源检索→证据压缩→生成假设→构建证据图→挖掘空白→设计实验→流式写报告
4. 在报告中点击引用标记 [sN] 可在右侧面板查看对应文献
5. 使用章节润色功能对某段进行扩写/缩写

### Demo 2：创造工坊 — 一句话生成医学报告助手

1. 进入创造工坊，点击「+ 创建功能」
2. 选择 AI Quick Build，输入："帮我做一个病历生成助手，输入患者症状后自动生成标准病历"
3. Hy3 30 秒内生成功能（emoji/名称/Prompt/医学双栏布局/引导语）
4. 点击进入功能独立工作区
5. 使用引导语开始对话，体验专属布局的医学助手
6. 退出后再次点击，确认可正常重新进入

### Demo 3（可选）：思路提炼 — 研究方向打磨

1. 进入思路提炼，输入："我想研究大模型的长上下文能力，有哪些值得做的方向"
2. Advisor 自动检索真实文献，带 [rN] 引用回答
3. 多轮对话后点击「生成指南」，输出思路收敛结果
